# Yorushika 演唱会票务辅助网站产品规划

## 1. 项目定位

本项目定位为“跨境演唱会票务辅助与履约协同平台”，服务对象是希望合规参与日本演唱会购票、支付、发券、出行与售后流程的中文用户。

平台的核心价值是：

1. 降低中文用户理解日本票务规则的门槛。
2. 为官方允许的购票路径提供流程可视化与客服协同。
3. 提供服务费、代办费、行程相关费用的国内支付能力。
4. 对接日本端工作人员处理线下支持事项，但不绕过官方实名、电子票、本人确认或平台限制。

## 2. 合规边界

以下能力不建议实现，也不应写入正式产品需求：

1. 伪造、随机分配用户收货或居住地址用于满足票务资格。
2. 为绕过平台限制而分配日本手机号、短信验证码或虚拟身份信息。
3. 在未获票务平台明确许可时，代替用户完成实名购票、抽选、电子票绑定或本人确认。
4. 抓取登录后页面、验证码页面、会员私域页面或规避 robots / ToS 的采集方式。
5. 以“代付票款”形式规避官方支付限制、会员体系限制或抽票规则。

可行替代方案：

1. 地址能力仅支持“用户真实入住地址”或“用户已预订酒店/民宿地址”上传与管理。
2. 平台仅收取服务费、咨询费、陪同费等自有服务费用，并通过微信/支付宝收款。
3. 官方票款支付只在目标票务平台允许的范围内处理，例如官方国际入口、官方支持的海外支付、或有书面授权的日本合作方。
4. 便利店付款场景仅作为“待办协同任务”管理，由在日工作人员线下执行，系统负责留痕、提醒、审计与回传结果。

## 3. 目标用户

### 核心用户

1. 不懂日语但想参加 Yorushika 日本演唱会的中国用户。
2. 需要客服陪跑的首次赴日观演用户。
3. 已有日本入住地址或出行安排，但缺少票务规则理解与支付协同能力的用户。

### 运营用户

1. 中文客服
2. 日本现场工作人员
3. 财务与风控人员

## 4. 产品目标

### 业务目标

1. 建立以 Yorushika 为核心的票务辅助 MVP。
2. 打通“资讯 -> 报名 -> 抽选任务 -> 支付协同 -> 发券 -> 出行支持”的闭环。
3. 沉淀日本票务平台规则知识库，便于后续扩展到更多艺人。

### 用户目标

1. 让用户清楚知道每一场公演的报名时间、规则、限制和当前状态。
2. 让用户可以用中文完成资料提交、客服沟通和订单追踪。
3. 让平台内部可以协同处理线下付款、资料核验和异常订单。

## 6. 核心功能设计

### 6.1 前台用户端

1. 演出列表页
   - 展示场次、时间、城市、场馆、票价、当前报名状态
2. 场次详情页
   - 聚合官方规则、报名时间线、风险提示、FAQ
3. 资格检查器
   - 用户选择“是否有官方会员 / 是否有本人护照 / 是否已有日本入住地址 / 是否能本人到场”
   - 输出可参加的官方通道与限制说明
4. 报名申请页
   - 提交姓名、证件、出行计划、实际入住地址、同行信息
5. 订单中心
   - 抽选任务、付款任务、发券状态、客服备注、异常记录
6. 服务费支付页
   - 微信 / 支付宝支付占位
7. 客服沟通页
   - FAQ、工单、进度回传、材料补交

### 6.2 运营后台

1. 演出源管理
   - 录入/同步官方场次、入口链接、规则快照
2. 用户订单看板
   - 待审核、待报名、待付款、待发券、已完成、异常
3. 便利店付款任务池
   - 指派在日工作人员
   - 记录付款截止时间、店铺、回执、付款状态
4. 风控审核
   - 核对护照名、拼音、生日、用户授权材料
5. 地址资料管理
   - 仅允许真实地址或真实订单地址
6. 财务对账
   - 服务费支付单
   - 人工代办成本单
   - 退款与补差记录

## 7. 支付方案建议

### 7.1 中国端

平台可收取以下费用：

1. 会员费
2. 服务费
3. 人工代办费
4. 出行协助费

接入方式建议：

1. 微信支付
2. 支付宝当面付 / 电脑网站支付 / 小程序支付

### 7.2 日本端

日本票款不建议设计成“绕规则代付”。更合规的处理模型：

1. 用户下单支付国内服务费。
2. 系统创建“官方票务支付任务”。
3. 若票务平台允许用户自行支付，则引导用户前往官方页面完成。
4. 若存在便利店付款且由在日工作人员执行，则要求：
   - 用户提交明确授权
   - 记录付款凭据与任务日志
   - 明确该操作受制于官方规则，不承诺突破限制

## 8. 地址方案建议

你提出的“随机分配场馆附近民宿或酒店地址”不建议作为产品功能，因为这会直接触碰资格规避风险。

建议改为：

1. 用户上传自己真实入住订单
2. 平台提供“场馆附近住宿推荐”
3. 用户从推荐列表中自主预订
4. 订单里只保存真实已预订地址

未来若要扩展，可新增：

1. OTA 链接聚合
2. 场馆周边住宿热力图
3. 入住日期与演出日期自动匹配

## 10. 规则采集与爬虫设计

### 10.1 采集原则

1. 只抓取公开页面
2. 保存原始快照
3. 提取结构化字段
4. 为每条规则打上来源 URL 和抓取时间

### 10.2 首期爬虫入口

1. Yorushika 官方新闻页
2. Yorushika 专题页
3. Lawson Ticket 艺人公开页
4. Lawson Ticket 公开售卖页
5. Tixplus 公开说明页

### 10.3 不采集内容

1. 登录后页面
2. 账户信息页
3. 抽选申请表单内部接口
4. 需要验证码验证的页面

## 11. MVP 范围

### MVP 必做

1. 官方场次聚合
2. 详情页与规则卡片
3. 资格评估器
4. 报名申请表单
5. 订单状态页
6. 服务费支付占位接口
7. 运营侧便利店付款任务池

### MVP 不做

1. 自动抢票
2. 自动提交抽选
3. 虚拟手机号分配
4. 虚假地址生成
5. 电子票破解或转绑

## 12. 里程碑

### Phase 1

1. PRD 定稿
2. 数据模型与原型页
3. 官方入口采集器

### Phase 2

1. 用户申请与订单系统
2. 支付占位
3. 运营后台最小版

### Phase 3

1. 规则快照比对
2. 通知中心
3. 场馆周边住宿推荐

## 13. 风险

1. 日本票务平台规则变化快，必须保留人工审核。
2. 电子票实名和照片核验会显著限制代办空间。
3. 若产品表述不清，容易被误解为代抢或规避实名平台。
4. 跨境支付链条涉及合规、退款、汇率和拒付风险。

## 14. 建议的产品表述

建议对外将产品描述为：

“面向中文用户的日本演唱会票务辅助与履约协同平台，提供官方信息聚合、资格判断、报名资料整理、客服协助、服务费支付及现场支持，不提供绕过票务平台规则的服务。”
